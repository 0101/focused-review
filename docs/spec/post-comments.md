# Post Comments to PR

Post focused-review findings as inline PR comments on GitHub and Azure DevOps, with user preview and approval before posting.

## Goals

1. Let users post review findings from a local `/focused-review` run back to the PR as inline comments
2. Support both GitHub (Pull Request Reviews API) and Azure DevOps (PR Threads API)
3. Require explicit user approval before posting (comments go under the user's name)
4. Add attribution footer linking to focused-review and showing the approver's name
5. Include findings outside the PR diff in the overall review body (not as inline comments)

## Expected Behavior

### Invocation

```
/focused-review post-comments <PR-URL>
```

The user has already run a review (`/focused-review branch` or similar) on a local checkout of the PR branch. A review report exists at `.agents/focused-review/review-*.md`.

### Flow

1. **Parse URL** — Python `parse-pr-url` extracts platform + identifiers from the PR URL
2. **Locate report** — find latest `review-*.md` in `.agents/focused-review/`; if multiple exist, ask user which one
3. **Get user identity** — Python `get-pr-user` fetches the authenticated user's name/login via `gh`/`az` CLI
4. **Read report & build comments** — the skill (LLM) reads the review report natively, extracts confirmed/questionable findings, reads the diff at `.agents/focused-review/diff.patch` to classify findings as in-diff or out-of-diff, formats comment bodies with severity emoji + description + suggestion + attribution footer, and writes `comments.json`
5. **Preview** — present findings table to user showing what will be posted and where
6. **User approval** — user approves all or excludes specific findings by number
7. **Post** — Python `post-comments` reads `comments.json` and posts via `gh api` / `az` CLI
8. **Report result** — show success/failure summary with links to posted comments

### Supported PR URL Formats

```
# GitHub
https://github.com/{owner}/{repo}/pull/{number}
https://github.com/{owner}/{repo}/pull/{number}/files
https://github.com/{owner}/{repo}/pull/{number}/commits

# Azure DevOps
https://dev.azure.com/{org}/{project}/_git/{repo}/pullrequest/{id}
https://{org}.visualstudio.com/{project}/_git/{repo}/pullrequest/{id}
```

### Comment Format

Inline comment body (per finding):
```markdown
### 🔴 [High] Null reference risk

`UserService.cs:42` — `GetUser()` can return null but caller doesn't check.

**Suggestion:** Add null check before accessing `user.Name`.

---
<sub>Generated by [focused-review](https://github.com/0101/focused-review), approved by @username</sub>
```

Severity emoji: 🔴 Critical/High, 🟡 Medium, 🔵 Low

Overall review body includes a summary table of all findings + any out-of-diff findings listed in full.

### `comments.json` Schema

Written by the skill, consumed by Python `post-comments`:

```json
{
  "platform": "github",
  "owner": "owner",
  "repo": "repo",
  "pr_number": 123,
  "review_body": "## Focused Review\n...",
  "inline_comments": [
    {
      "id": 1,
      "path": "src/foo.cs",
      "line": 42,
      "body": "### 🔴 [High] Null reference risk\n..."
    }
  ]
}
```

For ADO, additional fields: `org`, `project`. No `owner`.

### Platform API Details

**GitHub**: Single `POST /repos/{owner}/{repo}/pulls/{number}/reviews` call posts the review body + all inline comments atomically. Event: `COMMENT` (neutral — doesn't approve or reject).

**ADO**: One `POST .../_apis/git/repositories/{repo}/pullRequests/{id}/threads` call per inline comment (no batch endpoint). Overall comment is a thread without `threadContext`.

### Out-of-Diff Findings

Findings referencing lines not in the PR diff cannot be posted as inline comments. They are included in the overall review body with a "Findings outside PR diff" section.

### Error Handling

- CLI not installed → clear error: "Install `gh` CLI and run `gh auth login`"
- Not authenticated → "Run `gh auth login` first"
- Partial failure (ADO per-thread posting) → post what succeeds, report failures

## Key Design Decisions

- **No Python parsing of LLM output**: the skill reads the review report and writes `comments.json`. Python only handles mechanical tasks (URL parsing, CLI invocation, file I/O).
- **CLI tools for API calls**: `gh` for GitHub, `az` for ADO — zero new Python dependencies, auth handled by user's existing CLI login.
- **User approval required**: comments are posted under the user's name, so they must review and approve everything first.
- **Attribution footer**: every comment includes `Generated by [focused-review](...), approved by {username}`.

## Technical Approach

### New Python Subcommands (in `focused-review.py`)

All mechanical — no report parsing:

- `parse-pr-url --url <URL>` → JSON with platform, owner/org, repo, project (ADO), pr_number
- `get-pr-user --platform <github|ado>` → JSON with username, display_name
- `post-comments --comments <path> [--exclude N,N]` → posts via `gh api` or `az` CLI, outputs result JSON

### New Skill File

`skills/focused-review/POST-COMMENTS.md` — orchestration instructions for the LLM. Handles report reading, comment formatting, preview, approval, and writing `comments.json`.

### SKILL.md Routing

Add `post-comments` to the argument list in `SKILL.md` to route to `POST-COMMENTS.md`.
